## 4. 담당자7 협의 필요 — 공통코드 상세값 미확정

`AUCTION.AUC_STATUS_CD`(AUCG01), `BID.BID_STATUS_CD`(BIDG01)는 `08_DB_기초데이터`에 **그룹 코드만** 있고 상세코드(진행중/종료, 최고입찰/반환 등)는 아직 없습니다. 지금은 임시 상수(`AuctionStatusCode.IN_PROGRESS = "AUC_IN_PROGRESS"` 등)를 쓰고 있는데, 실제 `CMM_CODE` 값이 확정되면 이 상수들만 교체하면 되도록 한 곳에 모아뒀습니다.

- `nct.auction.constant.AuctionStatusCode` — IN_PROGRESS, ENDED
- `nct.auction.constant.BidStatusCode` — ACTIVE(최고입찰), OUTBID(반환됨)

**요청:** 실제 CMM_CD 값 확정해서 알려주시면 상수 값만 교체하겠습니다.

## 5. 다른 담당자와 협의가 필요한 계약(포트)

### 5.1 `ProductQueryPort` (담당자2)

```java
public interface ProductQueryPort {
    ProductBidInfo getBidInfo(Long prdSn); // sellerUsrSn, buyNowAmt
}
```
`PRODUCT` 테이블에서 판매자 회원번호와 즉시구매금액(`PRD_IBY_AMT`)만 필요합니다. 실제 구현체를 이 인터페이스로 만들어 `@Component` 등록해주시면 됩니다.

### 5.2 `PointLedgerPort` (담당자6) — 013 이후 2개 메서드 추가됨

```java
public interface PointLedgerPort {
    boolean hasAvailableBalance(Long usrSn, Long amount);          // 013부터 있던 것
    void holdForBid(Long usrSn, Long amount, Long bidSn);          // 신규 - F-AUC-015
    void releaseHold(Long usrSn, Long amount, Long bidSn);         // 신규 - F-AUC-016
}
```
**협의 필요 항목** (`PointLedgerPort.java` 주석에도 명시):
1. `bidSn`을 원장 참조키로 남길지 (반환 시점 추적용)
2. 홀딩/반환 실패 시 예외로 알릴지, `boolean`으로 알릴지
3. 이 메서드가 자체 트랜잭션인지, `BidService`의 트랜잭션에 참여(propagation)하는지 — **참여해야 "BID 기록 성공 + 포인트 홀딩 성공"이 원자적**이 됩니다. 저희 쪽 `executeBid`/`executeBuyNow`는 `@Transactional`이라, 이 메서드가 별도 트랜잭션(REQUIRES_NEW)이면 저희 쪽 롤백과 어긋날 수 있습니다.

### 5.3 `TradeCreationPort` (담당자4) — 신규 제안

```java
public interface TradeCreationPort {
    Long createTradeForBuyNow(Long aucSn, Long prdSn, Long sellerUsrSn, Long buyerUsrSn, Long finalAmt);
}
```
즉시구매(F-AUC-018) 성공 시 `TRADE` 생성을 위해 제안한 계약입니다. **09번 문서 10.1절 "TRADE 유형·참조 조합 검증"이 아직 담당자4 주도 미결정 항목**이라, 저희 쪽에서 확실히 아는 최소 정보(경매/상품/판매자/구매자/금액)만 넘기고 `TRD_TYPE_CD` 등 나머지는 담당자4 구현체가 결정하도록 설계했습니다.

**협의 필요 항목:**
1. `TRD_TYPE_CD` 값 (경매 낙찰과 즉시구매를 같은 코드로 볼지)
2. `TRADE`에 `AUC_SN`을 남길 컬럼이 필요한지 (현재 DDL엔 `PRD_SN`만 있음)
3. 이 메서드도 트랜잭션 참여 여부 협의 필요 (5.2-3과 동일한 이슈)

## 7. 담당자5에게 참고 — 낙찰 확정(F-AUC-020) 관련

- `내 입찰 내역 조회`(F-AUC-022)에서 "낙찰" 상태를 아직 `BID_STATUS_CD`가 표현하지 못해서, `MyBidHistoryItem.resolveDisplayStatus()`가 "경매 종료(ENDED) + 입찰 ACTIVE"를 잠정적으로 "WON"으로 해석하고 있습니다. F-AUC-020에서 전용 낙찰 상태를 확정하시면 이 잠정 로직을 그 상태값 참조로 단순화할 수 있습니다.
- 즉시구매(F-AUC-018)로 종료된 경매는 낙찰 확정 플로우(F-AUC-020)를 안 거치고 곧바로 `AUCTION.AUC_STATUS_CD = ENDED`, 낙찰자의 `BID`는 `ACTIVE` 상태로 남습니다. F-AUC-020 설계 시 "즉시구매로 이미 끝난 경매"는 낙찰 확정 대상에서 제외해야 합니다.

## 8. 담당자4에게 참고 — 거래 생성 연동 지점

`executeBid`(입찰), `executeBuyNow`(즉시구매) 둘 다 `AUCTION` 행에 `SELECT ... FOR UPDATE` 잠금을 걸고 하나의 트랜잭션 안에서 처리합니다. `TradeCreationPort.createTradeForBuyNow()`는 이 트랜잭션 안에서 호출되므로, 실제 구현체가 별도 트랜잭션으로 동작하면 "경매는 종료됐는데 거래는 안 만들어짐" 같은 불일치가 생길 수 있습니다 (5.3-3 참고).
